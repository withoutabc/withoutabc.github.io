# 一维数组 append 进二维数组

比较两种写法

在这两种情况下，最终得到的 res 切片中的内容是相同的，因为它们都包含了 path 切片的内容。然而，两者之间存在着一些微妙的差异：

  1. res = append(res, tmp) ： 这种方式创建了一个新的切片 tmp ，并将 path 的内容复制到了 tmp 中。然后将 tmp 添加到 res 中，因此 res 中的每个元素都是一个独立的切片，与原始 path 完全无关。
  2. res = append(res, path) ： 这种方式直接将 path 添加到 res 中，因此 res 中的元素与 path 共享相同的底层数组。这意味着，如果后续修改了 path 的内容， res 中相应位置的内容也会受到影响。

如果希望在 res 中保留 path 的原始内容，最好使用 res = append(res, tmp) 的方式。

# Docker-compose 部署挂载卷的问题

有时候在网上复制 docker-compose.yml 文件后遇到神奇报错。

docker-compose.yml 文件中定义的 mysql_data 卷是一个匿名卷。匿名卷是通过在 volumes 部分定义的卷，而没有显式指定卷名称的方式创建的。在你的配置中， mysql_data 是一个匿名卷，因为它没有指定一个明确的卷名称。

匿名卷的特点是：

  • 它们的名称是自动生成的，通常以一个随机字符串开头，例如： <项目名称>_mysql_data .
  • 它们不会持久化保存在 Docker 主机上的文件系统中，而是将数据存储在 Docker 卷管理系统中。这意味着它们可能会在容器删除后仍然存在，但不再关联到任何容器。
  • 当你删除拥有匿名卷的容器时，卷本身并不会被自动删除，你需要手动删除这些卷。

在你的配置中，当你运行 docker-compose down 命令停止并删除容器时， mysql_data 卷也将保留下来。如果你想要在删除容器时同时删除这个卷，你可以在 docker-compose down 命令后面添加 --volumes 参数：

这将删除拥有匿名卷的容器，并且同时删除这些卷。请确保在删除卷之前已经备份了重要的数据。

如果忘记附带 --volumes 停止容器，再修改 docker-compose.yml 可能会出问题。这时候最好把对应的匿名卷。因为再次启动 docker-compose.yml 时，会用之前保存的数据启动，可能镜像或版本号改了导致出问题。
